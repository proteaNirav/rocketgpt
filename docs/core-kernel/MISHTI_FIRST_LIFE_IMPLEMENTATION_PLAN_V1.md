# Mishti First Life Implementation Plan V1

## 1. Purpose

This document defines the minimum implementation sequence required to move Mishti AI from architecture to first governed operational life.

## 2. First-Life Definition

First life means Mishti AI can:

- operate continuously in a bounded way
- execute governed tasks
- route work through trusted builders
- preserve evidence continuity
- enter degraded or kill states safely
- remain under owner-held root authority

It does not mean final-scale autonomy.

## 3. Mandatory Foundations

The following must exist before first life:

1. constitution and zero-trust baseline
2. owner-held root authority and key model
3. emergency kill and degraded-state controls
4. minimal governance enforcement path
5. minimal Brain task derivation and routing path
6. Librarian registry and routing index
7. Documentor evidence continuity path
8. at least one safe execution path through builder -> runtime

## 4. Recommended Implementation Order

### Phase 1. Control Foundations

- ratify Constitution V1
- establish root authority and key custody
- define zero-trust and kill-state controls

### Phase 2. Governance Kernel

- stand up minimal Governance Engine
- enable proposal, policy, emergency, and rollback control surfaces

### Phase 3. Registry And Evidence

- stand up Librarian registry
- stand up Documentor Mesh intake and anchoring path
- require evidence for protected actions

### Phase 4. Brain And Task Core

- implement bounded goal intake
- implement constrained task derivation
- implement builder routing through trust-aware registry

### Phase 5. Builder And Runtime Path

- register first trusted builders
- enable first safe runtime execution path
- enable result verification and output evidence

### Phase 6. Survival And Recovery

- validate degraded mode
- validate kill path
- validate rollback and recovery-only paths

## 5. Minimum Viable Live System

The Minimum Viable Live System should include:

- one owner-controlled root authority path
- one governance authority backend
- one Librarian registry path
- one evidence anchoring path
- one Brain planning and routing path
- one or more trusted builders
- one safe runtime execution path
- one emergency kill path outside normal autonomy flow

## 6. Items That Can Wait

The following may wait until after first life:

- large-scale distributed optimization
- deep autonomous research loops
- broad self-improvement automation
- multi-region scale-out
- rich analytics beyond essential survival and routing metrics

## 7. Exit Criteria

Mishti AI is ready for first life when:

- protected actions are governed
- authority is scoped and revocable
- builders are trust-routed
- evidence continuity exists
- degraded and kill states work
- recovery-only mode works

## 8. Principle

The first-life plan prioritizes survivable bounded operation over feature breadth. Mishti AI should become live only after control, evidence, and recovery exist.
